# Vibe Coding 中文标准工作流

## 使用目标

本 Skill 用于指导 Agent 以标准工程流程推进项目，而不是直接一次性生成代码。目标是将一个模糊想法、考核题目或已有仓库，整理为具有明确需求、技术方案、实现计划、项目记忆、测试验证、版本回退和交付文档的完整项目。

核心原则：

- 先文档，后代码。
- 先最小可运行版本，后功能扩展。
- 一次只执行一个可验证步骤。
- 每个实现步骤必须有验证方式。
- 持续维护 `memory-bank/`，使其成为长期事实来源。
- 每次重大变更后更新进度记录和架构记录。
- 出现问题时先判断最小修复或版本回退，不盲目连续修改。

## 适用场景

在以下情况使用本 Skill：

- 用户需要完成一个完整项目或项目原型。
- 用户需要参加 AI 工具使用考核或课程设计。
- 用户需要从零启动一个机器人、Web、数据分析、AI 应用、自动化脚本或可视化项目。
- 用户已有一个混乱仓库，需要规范化为文档先行的开发流程。
- 用户需要通过 Agent/Codex/Claude Code 等工具分步完成项目交付。
- 用户需要保留开发记录、版本回退路径、技术报告和答辩材料。

## 核心产物

项目应创建并持续维护以下文件：

```text
PRD.md
memory-bank/design-document.md
memory-bank/tech-stack.md
memory-bank/implementation-plan.md
memory-bank/progress.md
memory-bank/architecture.md
AGENTS.md 或 CLAUDE.md
README.md
changelog.md
```

文件职责：

- `PRD.md`：临时想法捕获文件，仅用于项目启动阶段。
- `memory-bank/design-document.md`：正式设计文档，记录范围、非目标、用户流程、功能行为、边界情况和验收标准。
- `memory-bank/tech-stack.md`：技术选型文档，记录语言、框架、第三方库、开发工具、运行环境和测试方式。
- `memory-bank/implementation-plan.md`：实现计划，只写任务，不写代码；每一步必须小、具体、可验证。
- `memory-bank/progress.md`：执行进度日志，记录已完成步骤、验证结果和当前状态。
- `memory-bank/architecture.md`：项目架构记录，维护重要文件、模块职责、调用关系和重大变更。
- `AGENTS.md` / `CLAUDE.md`：Agent 工作规则，约束写代码前读取项目记忆库。
- `README.md`：项目说明文档。
- `changelog.md`：版本变化记录，可选但推荐。

`PRD.md` 在项目初期用于承接想法。项目进入正式开发后，`memory-bank/` 是长期事实来源。

## 标准工作流

### 1. 项目初拟

将用户的模糊想法整理为可评估的项目方向。

必须明确：

- 项目名称。
- 目标用户。
- 用户问题。
- 应用场景。
- 核心功能。
- 推荐语言和技术栈。
- 最小可运行版本范围。
- 最终展示效果。
- 主要约束和合理假设。

当用户未确定方向时，给出 3 个以内可行方案，并推荐最稳妥、最容易跑通、最适合展示的方案。

### 2. 生成 PRD.md

创建轻量级 `PRD.md`，用于捕获初始想法。

`PRD.md` 应包含：

- 产品 / 项目名称。
- 目标用户。
- 用户问题。
- 项目目标。
- 主要用户流程。
- 核心功能。
- 关键约束。
- 当前假设。
- 最小可运行版本范围。
- 暂不实现内容。
- 待澄清问题。

保持简洁，不在 PRD 阶段写成完整技术方案。

### 3. 需求澄清

读取 `PRD.md` 后，判断需求是否足以进入正式设计。

重点检查：

- 项目范围是否明确。
- 非目标范围是否明确。
- 用户流程是否完整。
- 功能行为是否可实现。
- 输入输出是否清楚。
- 数据或状态模型是否明确。
- 异常情况是否覆盖。
- 验收标准是否可验证。

若仍有歧义，提出必要且集中的澄清问题。若用户要求快速推进，可基于合理假设继续，但必须记录假设。

### 4. 生成 design-document.md

将澄清后的需求沉淀为 `memory-bank/design-document.md`。

内容应至少包括：

- 项目背景。
- 项目目标。
- 项目范围。
- 非目标范围。
- 用户流程。
- 功能行为。
- 数据 / 状态模型。
- 边界情况。
- 异常处理。
- 验收标准。

设计文档必须足以支撑后续技术选型和实现计划。

### 5. 生成 tech-stack.md

根据 `design-document.md` 生成 `memory-bank/tech-stack.md`。技术选型必须来自需求，不追求无必要的新技术。

必须明确：

- 编程语言及选择理由。
- 应用框架或不使用框架的理由。
- 前端、后端、算法、数据、可视化、存储、部署等层的技术选择。
- 第三方库及用途。
- 开发工具。
- 运行环境。
- 安装方式。
- 启动方式。
- 测试方式。
- 可替代方案及不选择原因。

优先选择简单、稳定、易实现、易展示的技术。

### 6. 建立 memory-bank

创建并维护 `memory-bank/`。

建议结构：

```text
memory-bank/
├── design-document.md
├── tech-stack.md
├── implementation-plan.md
├── progress.md
└── architecture.md
```

规则：

- `design-document.md` 记录产品和需求事实。
- `tech-stack.md` 记录技术事实。
- `implementation-plan.md` 记录可执行任务序列。
- `progress.md` 只记录执行日志，不承载设计职责。
- `architecture.md` 记录当前项目文件地图和模块职责。
- 每次重大功能完成后更新 `progress.md` 和 `architecture.md`。

### 7. 生成 Agent 工作规则

在 `AGENTS.md` 或 `CLAUDE.md` 中写入以下规则。若两者同时存在，保持内容一致。

```md
## 重要提示

- 写任何代码前必须完整阅读 `memory-bank/architecture.md`
- 写任何代码前必须完整阅读 `memory-bank/design-document.md`
- 写任何代码前必须完整阅读 `memory-bank/tech-stack.md`
- 写任何代码前必须完整阅读 `memory-bank/implementation-plan.md`
- 写任何代码前必须完整阅读 `memory-bank/progress.md`
- 每完成一个重大功能或里程碑后，必须更新 `memory-bank/progress.md`
- 每完成一个重大功能或里程碑后，必须更新 `memory-bank/architecture.md`
- 不允许跳过当前步骤直接执行下一步骤
- 不允许在未验证当前步骤的情况下继续扩展新功能
```

### 8. 生成 implementation-plan.md

根据 `design-document.md` 和 `tech-stack.md` 生成 `memory-bank/implementation-plan.md`。

实现计划必须满足：

- code-free，只写任务，不写代码。
- 步骤小、具体、有顺序。
- 每一步可独立执行。
- 每一步可独立验证。
- 每一步有明确完成标准。
- 先覆盖最小可运行版本，再安排增强功能。
- 避免“大而空”的任务描述。

每个步骤应包含：

- 步骤编号。
- 步骤目标。
- 涉及文件或模块。
- 具体任务。
- 不做什么。
- 验证方式。
- 完成标准。

禁止使用以下模糊表达作为独立步骤：

- 完成后端。
- 完成前端。
- 完成所有页面。
- 完善功能。
- 优化项目。
- 写完测试。

### 9. 功能板块与模块分类

在写代码前完成板块和模块拆分。

功能板块应区分：

- 核心板块。
- 辅助板块。
- 展示板块。
- 测试板块。
- 文档板块。

每个板块记录：

- 作用。
- 输入。
- 处理逻辑。
- 输出。
- 依赖关系。
- 是否属于最小可运行版本。

模块分类应输出：

- 项目目录树。
- 文件或组件职责。
- 输入输出。
- 关键函数或类。
- 模块调用关系。
- 是否需要测试。
- 推荐开发顺序。

### 10. 执行单步实现

执行阶段必须一次只做 `implementation-plan.md` 中的一个步骤。

写代码前必须重新读取：

```text
memory-bank/design-document.md
memory-bank/tech-stack.md
memory-bank/implementation-plan.md
memory-bank/progress.md
memory-bank/architecture.md
```

执行规则：

- 只执行当前步骤。
- 不提前实现下一步骤。
- 写代码前说明当前步骤目标、涉及文件和验证方式。
- 代码必须完整可运行，不只给伪代码。
- 新增文件时说明原因。
- 修改文件时说明修改原因。
- 完成后给出运行命令或测试方式。
- 当前步骤未验证前，不进入下一步骤。

### 11. 单步验证

每完成一个步骤后进行验证。

验证内容：

- 当前步骤是否完成。
- 是否满足验收标准。
- 是否与设计文档一致。
- 是否影响其他模块。
- 是否需要更新 `progress.md`。
- 是否需要更新 `architecture.md`。

通过后：

- 向 `memory-bank/progress.md` 追加简洁记录。
- 更新 `memory-bank/architecture.md` 中的新增文件、修改文件和职责变化。
- 必要时创建 Git checkpoint。

未通过时：

- 说明问题。
- 给出最小修复方案。
- 判断是否需要版本回退。

### 12. 历史记录与版本回退

优先使用 Git 管理历史记录。若环境不支持 Git，则使用手动备份目录。

推荐版本节点：

```text
v0.1 项目结构初始化
v0.2 最小功能跑通
v0.3 核心交互或可视化完成
v0.4 测试通过
v1.0 最终展示版本
```

推荐 Git 操作：

```bash
git init
git add .
git commit -m "chore: initialize project structure"
git status
git log --oneline
git checkout -b feature/<feature-name>
git checkout main
git merge feature/<feature-name>
git tag v0.1
```

回退原则：

- 小范围错误优先最小修复。
- 多文件混乱或稳定版本被破坏时优先回退。
- 回退前记录当前问题和最近修改。
- 回退后先验证稳定版本，再重新尝试功能修改。

常用回退命令：

```bash
git restore <file>
git restore .
git reset --hard HEAD
git checkout <commit-id>
```

谨慎使用 `git reset --hard`，执行前确认是否需要保留未提交修改。

### 13. 报错修复

报错处理流程：

1. 识别报错类型：语法、导入、变量、类型、依赖、路径、环境、逻辑或集成错误。
2. 定位文件、函数和行号。
3. 判断是否由最近修改引入。
4. 优先给出最小修复方案。
5. 判断是否需要大范围重构。
6. 判断是否需要回退版本。
7. 给出修复后的验证命令。
8. 更新 `progress.md`，必要时更新 `architecture.md`。

禁止在未定位原因前连续大改代码。

### 14. 模块测试

每个核心模块完成后进行单元或模块测试。

测试内容：

- 模块功能概述。
- 输入和输出。
- 正常测试用例。
- 异常测试用例。
- 边界测试用例。
- 自动化测试方式。
- 运行命令。
- 通过标准。

优先使用项目技术栈中简单稳定的测试工具，例如 Python 项目可使用 `pytest` 或 `unittest`。

### 15. 集成测试

多个模块完成后进行集成测试。

重点检查：

- 模块调用关系。
- 文件导入路径。
- 参数传递。
- 数据格式。
- 主流程完整性。
- 循环依赖。
- 命名一致性。
- 异常情况。
- 与 `design-document.md` 的一致性。

集成测试必须给出测试流程、测试用例、重点接口、问题清单、修复建议和最终验收标准。

### 16. 项目优化

最小可运行版本通过后，再进行优化。

优化优先级：

1. 必须优化：影响稳定性、运行、验收的问题。
2. 推荐优化：明显提升展示效果和用户体验的问题。
3. 有时间再优化：锦上添花但非必要功能。
4. 当前不建议优化：复杂度高、风险高、收益不明确的功能。

优化方向：

- 功能增强。
- 交互优化。
- 可视化优化。
- 日志完善。
- 异常处理。
- 参数配置。
- 测试完善。
- 文档完善。
- 性能优化。
- 答辩展示亮点。

### 17. 文档整理

项目交付前整理文档。

至少包括：

- `README.md`
- 使用说明
- 测试说明
- 技术报告
- `changelog.md`
- `memory-bank/progress.md`
- `memory-bank/architecture.md`
- 答辩稿
- FAQ

文档要求：

- 简洁。
- 可提交。
- 可复现。
- 能说明使用什么语言、什么技术、什么依赖、如何运行、如何测试。
- 能体现 AI 工具在项目流程中的作用。

### 18. 技术报告与答辩材料

技术报告应包含：

- 项目背景。
- 项目目标。
- 需求分析。
- 技术选型。
- 系统设计。
- 模块设计。
- 核心技术或算法。
- 实现过程。
- 测试方案与结果。
- AI 工具使用过程。
- 项目亮点。
- 不足与改进。
- 总结。

答辩材料应包含：

- 3 分钟讲稿。
- 5 分钟讲稿。
- 展示流程。
- 项目亮点话术。
- 常见问题与回答。

答辩表述重点：

> 我不是简单让 AI 替我写代码，而是使用 AI 完成了从项目初拟、需求分析、技术选型、模块拆分、代码实现、调试修复、模块测试、集成测试、版本记录、文档整理到技术报告撰写的完整工程流程。

## 非协商规则

- 不在需求不清楚时直接写代码。
- 不一次性生成整个项目。
- 不跳过设计文档和技术选型。
- 不让 `implementation-plan.md` 包含代码。
- 不执行未验证的下一步。
- 不把 `progress.md` 写成设计文档。
- 不让 `architecture.md` 过期。
- 不只保留最终代码，必须保留进度、架构、变更和回退记录。
- 不在代码出错后盲目连续修改，必须先判断最小修复还是版本回退。
- 不引入超出最小可运行版本太多的复杂功能。

## 最小交付标准

项目至少达到以下状态才算完成：

- `memory-bank/design-document.md` 已完成。
- `memory-bank/tech-stack.md` 已完成。
- `memory-bank/implementation-plan.md` 已完成，且无代码内容。
- `memory-bank/progress.md` 记录了主要执行过程。
- `memory-bank/architecture.md` 记录了当前文件结构和模块职责。
- 最小可运行版本可以启动。
- 核心功能通过验证。
- README 能说明安装、运行和测试方式。
- 有基本测试记录。
- 有项目总结或技术报告。
- 有版本回退或备份方案。

## 执行风格

- 使用中文。
- 输出简洁、明确、可执行。
- 优先表格和清单，但避免过度碎片化。
- 对用户已给出的信息不要重复追问。
- 信息不足时先提出关键问题；时间紧时基于假设推进。
- 始终优先稳定、可运行、可展示。
